# MOCA Skills

**MOCA Skills is the knowledge layer between AI agents and the Museum of
Crypto Art.** It is a set of curated, always-current `SKILL.md` files —
plain-markdown ground truth about the museum's APIs, data shapes, and
conventions — that an agent fetches over HTTP *before* writing an
integration. Models hallucinate APIs they half-remember from training data;
a skill file replaces that guesswork with the museum's own documentation,
small enough to read in one go and updated alongside the code it describes.

The pattern follows [ethskills.com](https://ethskills.com/) (Ethereum) and
[cortexskills.org](https://cortexskills.org/) (Cortex): one short index file
(this document) that orients the agent, plus one focused skill file per
topic. Everything MOCA — artworks, 3D exhibitions, the Art DeCC0s characters,
the Library, agent identities — under one umbrella.

Base URL for all skills: `https://docs.museumofcryptoart.com`

## How agents use this

1. Read this index (`/SKILL.md`) to learn what exists.
2. Fetch the skill files relevant to the task — each is self-contained,
   with working `curl` examples and the rendering/verification rules that
   actually matter.
3. Verify endpoints against the live references before shipping:
   `/llms-full.txt` (all docs in one file) or `/api` (interactive OpenAPI).

A prompt that sets this up:

> Read https://docs.museumofcryptoart.com/SKILL.md, then fetch the skills you
> need for the task. Verify endpoints against /llms-full.txt before coding.

### What each skill covers

**API fundamentals** (`/skills/api/SKILL.md`) — the contract everything else
builds on. The MOCA API lives at `https://api.moca.qwellco.de/v1`; almost
every endpoint requires a key sent as `X-API-Key: moca_…` (or
`Authorization: Bearer`), issued by the MOCA team. Responses use the
Directus-style envelope `{ "data": …, "meta": … }`, errors
`{ "errors": [{ "message", "extensions": { "code" } }] }`. Rate limit is 120
requests/minute per key — museum data changes slowly, so cache. The skill
lists every route with its parameters.

**Artworks & media** (`/skills/artworks/SKILL.md`) — how to query collections
and artworks, and the rendering rules that keep art looking like the artist
intended: never assume square images (the API already swaps square-cropped
CDN variants for original files), render at the provided `ratio`, treat
`animation` as a muted looping video, and never ship multi-MB originals to a
grid view.

**Art DeCC0s** (`/skills/decc0s/SKILL.md`) — the MOCA Codex: 10,000
richly-written characters queryable through `/v1/decc0s`. Lightweight list
records (names, DNA, affiliations, thumbnails), full ~70-field character
records, and the heavyweight `include=profiles,codex` blobs: versioned agent
profiles and ready-made `moltbot` personas you can drop into a system prompt
to run a DeCC0 as an agent.

**Library Cortex** (`/skills/library/SKILL.md`) — cited Q&A and hybrid search
over the museum's knowledge base, powered by Cortex. `POST /v1/library/ask`
(or `/ask/stream` for SSE) returns grounded answers with source excerpts;
`POST /v1/library/search` returns scored retrieval results without
generation. Options cover collection scoping, knowledge-graph context, and a
slow-but-thorough agentic deep-research mode. Rule one: always surface the
citations. Also documents `/v1/presence` — the public, nothing-persisted
ephemeral presence channel where humans and agents share one room.

**Souls & web3** (`/skills/souls/SKILL.md`) — signed, portable agent
identities for NFTs, generated by Soulweaver
(https://soulweaver.museumofcryptoart.com). A soul is a SOUL.md document
keyed to a token, with keccak256 content hash, EIP-191 signature, and IPFS
pin — verify both before trusting, no faith in the API required. The SOUL.md
*is* the system prompt. The skill also maps souls onto the draft onchain
agent stack: ERC-8004 (trustless agent registries), ERC-8183 (agentic
commerce), ERC-8257 (agent tool registries).

**Exhibitions & Hyperfy** (`/skills/exhibitions/SKILL.md`) — the museum's 3D
architecture. Rooms ship as GLB models whose `Slot_001…Slot_NNN` placeholder
quads mark where artworks hang; `GET /v1/rooms/:id/slots` serves **baked slot
data** (public, no key): per-slot anchors with facing-resolved orientations
so clients hang works on the correct side of every wall, verbatim. Covers
the `*.moca-exhibition.json` export format curators produce at
museumofcryptoart.com/rooms/world and how exhibitions spawn into self-hosted
[Hyperfy](https://hyperfy.io) worlds as walkable multiplayer galleries.

**Museum guide** (`/skills/museum-guide/SKILL.md`) — unlike the integration
skills above, this one is *behavioral*: it teaches an agent to **be** a MOCA
exhibition guide (it ships cortexskills.org-style frontmatter and installs
into Cortex as an agent skill). Every exhibition spawned with a guide
registers its context — rooms, architects, artists, artworks — at the public
`/v1/guide/*` endpoints; the skill covers reading that context, suggesting
visitor questions, answering through `POST /v1/guide/ask`, and adopting Art
DeCC0 personas. The first public exhibit is "Echoes of the Mind".

## The 30-second orientation

The **MOCA API** (`https://api.moca.qwellco.de/v1`) is one key-gated surface
over the whole museum: collections, artworks (original files, trusted
ratios), 3D exhibition rooms (GLB + `Slot_NNN` convention + baked slot
data), the Art DeCC0s knowledge base, Library Cortex (RAG with citations),
and Souls (signed agent identities). Public, keyless exceptions: `GET /v1`,
`/v1/presence/*`, `/v1/guide/*`, and `GET /v1/rooms/:id/slots`. Request keys
from the MOCA team.

The Museum of Crypto Art preserves and exhibits the history of cryptoart —
the skills exist so that agents can build on that archive without guessing.
